Claude Code 프롬프트: Animation Notify 처리 로직 조사
작업 유형: 조사 및 진단 문서 작성 (구현 변경 금지)
이번 작업은 진단/계획 단계입니다. 코드를 수정하지 말고, 실제 코드를 스캔한 결과만으로 진단 문서를 작성하세요. 문서 매핑이나 추측에 의존하지 말고 반드시 실제 소스 파일을 열어 확인할 것.
목적
Animation Notify 시스템이 "구조체 정의 → Core 처리 로직 → 실제 호출 지점"으로 이어지는 흐름에서 처리 로직이 어떻게 분포되어 있는지 파악한다. 처리 로직이 한 곳에 응집되어 있는지, 여러 파일에 흩어져 있는지, 흐름이 끊기는 지점은 없는지를 확인하는 것이 목표다.
조사 범위

Engine/Asset/Animation/Notify/ — Notify 관련 기본 구조체 정의를 모두 식별
Engine/Asset/Animation/Core/ — Notify 처리 로직(평가/트리거/디스패치 등)이 어느 파일·클래스·함수에 분포하는지 추적
호출 지점 — Core의 Notify 처리 함수를 실제로 호출하는 viewer / component 측 코드까지 추적

조사 항목

Notify 구조체 인벤토리

Notify/ 디렉토리의 각 파일에 정의된 구조체/클래스 이름과 역할
상속 관계 및 멤버 데이터 (예: 트리거 시간, 지속 구간, payload 등)
단발성(instant) Notify와 구간형(state/window) Notify의 구분이 타입 수준에서 어떻게 표현되는지


Core 내 처리 로직 분포

Notify를 참조·평가·트리거하는 함수가 어느 파일의 어느 함수에 있는지 (파일명 + 함수 시그니처 + 줄 범위)
처리 단계별 분류: (a) 애니메이션 평가 중 어떤 Notify가 발생했는지 판정하는 부분, (b) 발생한 Notify를 디스패치/콜백하는 부분, (c) 구간형 Notify의 begin/tick/end 상태 전이를 관리하는 부분
시간 진행(델타 타임) 기준으로 이전 프레임 시간 → 현재 프레임 시간 사이 구간을 어떻게 다루는지 (특히 looping/wrap-around 처리 여부)


호출 지점

Core의 Notify 처리 진입점을 호출하는 viewer/component 코드 위치
호출 경로가 애니메이션 평가 경로와 일관되게 연결되어 있는지


흐름 분석 및 진단

"구조체 정의 → Core 처리 → 호출"의 연결이 끊기는 지점이 있는지 (정의만 되어 있고 처리되지 않는 구조체, 호출되지 않는 처리 함수 등)
처리 로직이 흩어져 있다면 어느 파일들에 어떻게 분산되어 있는지, 응집도 관점에서의 문제점
